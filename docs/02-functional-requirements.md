# 機能要件定義書 — 太陽光卸・二次店営業管理 SaaS

本書は `docs/01-business-requirements.md`（業務要件定義書）と `product-proposal.md`（一次資料）を起点に、本 SaaS が提供する機能の入出力・受け入れ基準・関連ロール・非機能要件を定義する。後続の `tech-selection` / `ui-design` / `program-design` は本書の機能 ID (`F-001`〜) を相互参照キーとして利用する。

- 主要利用ロール: `saas_admin` / `wholesaler_admin` / `wholesaler_event_team` / `wholesaler_call_team` / `wholesaler_direct_sales` / `wholesaler_field_staff` / `dealer_admin` / `dealer_staff`
- 機能 ID は本書内連番 `F-001` から欠番なし。提案書側機能 ID (`F-001`〜`F-024` / `F-101`〜`F-111` / `F-201`〜`F-204`) は「関連業務要件 ID」フィールドで参照する。
- 優先度: `P0` = MVP 必須 / `P1` = MVP に含めるが薄実装可 / `P2` = Phase 2 以降
- 対応フェーズ: `Phase 1` (MVP, 〜2 か月) / `Phase 2` (LINE・CSV・BI 強化) / `Phase 3` (施工業者向け / 補助金詳細) / `Phase 4` (請求・決済外部連携)

## 2. 機能詳細

各機能は以下のテンプレートで記述する。

- **目的**: docs/01 のどの章・どの業務課題に対応するか
- **入力**: 主要画面の入力フィールドまたは API リクエスト
- **処理**: バリデーション・計算・状態遷移の要点
- **出力**: 画面表示・通知・永続化結果
- **受け入れ基準**: テスト可能な文（Given-When-Then 風）
- **関連ロール**: 操作可能なロール
- **関連業務要件 ID**: 提案書側機能 ID + docs/01 章番号

### 2.1 認証・テナント・ユーザー管理

#### F-001 サインイン

- **目的**: 業務時間帯 (8:00–22:00 JST) に全ロールが業務システムを利用できるようにする。docs/01 §8.4 セキュリティ。
- **入力**: メールアドレス、パスワード、（必須化されている場合）2FA コード
- **処理**: 認証成功でセッション発行。失敗 5 回で 15 分ロック。2FA 必須ロールは TOTP 検証を強制。多対多の二次店ユーザーは「現在の卸業者コンテキスト」をセッション内で保持。
- **出力**: ロール別ダッシュボードへ遷移
- **受け入れ基準**:
  - 正しい資格情報 + 2FA でログインできる
  - 失敗 5 回で 15 分の連続失敗ロック画面が出る
  - 二次店ユーザーが複数の卸業者と関係を持つ場合、ログイン直後に「卸業者選択」が表示される
- **関連ロール**: 全ロール
- **関連業務要件 ID**: docs/01 §8.4、提案書 §13.1-1

#### F-002 二段階認証 (2FA)

- **目的**: docs/01 §8.4 — 2FA を必須化できる設定を持つ
- **入力**: TOTP（Authenticator アプリ）QR スキャン → 検証コード
- **処理**: ロールに紐づく必須/任意ポリシーを `wholesaler_admin` が変更可能。`saas_admin` と `wholesaler_admin` は MVP で必須固定。バックアップコード 8 個を発行。
- **出力**: 2FA 設定済みフラグの永続化、ログイン時の追加チャレンジ
- **受け入れ基準**:
  - 必須ロールが 2FA 未設定でログインすると、強制的に 2FA 設定画面へ誘導される
  - バックアップコード 1 個を使うと、その 1 個は即座に無効化される
- **関連ロール**: 全ロール（`saas_admin`/`wholesaler_admin` 必須）
- **関連業務要件 ID**: docs/01 §8.4、Open Q4

#### F-003 パスワードリセット

- **目的**: 忘却時の自己復旧手段
- **入力**: 登録メールアドレス
- **処理**: 30 分有効なワンタイムリンクをメール送信。リンク先で新パスワード設定 + 2FA 再確認。
- **出力**: メール送信、パスワード更新、監査ログ
- **受け入れ基準**: リンクは 1 回限り、30 分超過で失効
- **関連ロール**: 全ロール
- **関連業務要件 ID**: docs/01 §8.4

#### F-004 卸業者テナント作成

- **目的**: 卸業者の招待制サインアップ（docs/01 §3.2）
- **入力**: 卸業者名、契約プラン、全体管理者の氏名・メール
- **処理**: テナント (`wholesaler` 型) + 全体管理者ユーザー作成 → 初期ログイン用招待メール送信
- **出力**: テナント ID、`wholesaler_admin` アカウント、初期招待コード
- **受け入れ基準**:
  - 同一メールでの全体管理者重複作成は拒否される
  - 招待メールから 7 日以内にアカウント開設しなかった場合、再発行できる
- **関連ロール**: `saas_admin`
- **関連業務要件 ID**: docs/01 §3.2、提案書 §13.1-2

#### F-005 プラン管理・請求状況確認

- **目的**: docs/01 §8.5 — 規模別プラン制（小・中・大）、運営側でのプラン変更・請求状況把握
- **入力**: テナント ID、プラン種別、ステータス（請求は外部運用 / オフライン記録）
- **処理**: プラン変更履歴を残す。請求書発行・決済は本システム外。
- **出力**: プラン台帳、請求状況一覧
- **受け入れ基準**: プラン変更時に変更履歴と監査ログが残る
- **関連ロール**: `saas_admin`
- **関連業務要件 ID**: docs/01 §8.5、Open Q1

#### F-006 卸業者内ユーザー招待・ロール管理

- **目的**: docs/01 §3.2 — 卸業者は配下要員をメール招待で追加
- **入力**: 招待先メール、ロール（`wholesaler_event_team` 等）
- **処理**: 招待メール送信 → アカウント作成。アクティブ／停止フラグ管理。
- **出力**: ユーザー台帳、ロール変更履歴
- **受け入れ基準**:
  - 全体管理者以外のロールは複数 (例: 催事 + コール) の付与不可とし、必要なら全体管理者を別途付与する（運用簡素化）
  - 停止ユーザーは即座にログイン不可になる
- **関連ロール**: `wholesaler_admin`
- **関連業務要件 ID**: docs/01 §3.2、提案書 §4

#### F-007 二次店セルフサインアップ

- **目的**: docs/01 §3.2 — 二次店はセルフサインアップ可、招待コードで卸業者に接続
- **入力**: 二次店名、代表メール、パスワード、招待コード
- **処理**: 招待コード検証 → 二次店テナント作成（既存二次店であれば追加関係を作成）→ `dealer_admin` ロール発行 → 卸業者-二次店 関係 (`relationship`) 作成
- **出力**: 二次店テナント ID、関係 ID、初回 2FA セットアップ画面
- **受け入れ基準**:
  - 同一二次店アカウントが複数の卸業者と関係を持てる（多対多）
  - 招待コードは有効期限・回数上限つきで卸業者側が発行・失効可能
- **関連ロール**: 新規二次店、`dealer_admin`
- **関連業務要件 ID**: docs/01 §3.2 / §7.5、提案書 §13.1-2

#### F-008 二次店内ユーザー管理

- **目的**: 二次店内の `dealer_staff` 追加・停止
- **入力**: 招待先メール、ロール
- **処理**: 招待メール送信。ロールは `dealer_admin` または `dealer_staff`。
- **出力**: ユーザー台帳、変更履歴
- **受け入れ基準**: 停止ユーザーは即座にログイン不可
- **関連ロール**: `dealer_admin`
- **関連業務要件 ID**: docs/01 §3.2

#### F-009 卸業者-二次店 関係管理

- **目的**: docs/01 §7.5 — 多対多テナント。スコープ・インセンティブ率・成績は卸業者ごとに独立
- **入力**: 関係先二次店（招待コード経由で確立）、関係ステータス（有効/停止）
- **処理**: 関係 ID を生成。これが本システムのテナント分離キー。関係単位で「業務スコープ」「インセンティブ率」「メモ」を保持。
- **出力**: 関係台帳、関係履歴
- **受け入れ基準**:
  - 関係を停止すると、その関係配下の希望提出・配属イベント・契約閲覧が二次店側で不可となる（既存の確定インセンティブは保持）
  - 関係 ID なしで他テナントデータにアクセスする API は 403 を返す
- **関連ロール**: `wholesaler_admin`
- **関連業務要件 ID**: docs/01 §7.5、提案書 §4

#### F-010 二次店業務スコープのデフォルト設定

- **目的**: docs/01 §3.3 / §7.6 — 「アポ獲得まで / 初回訪問まで / 商談・クロージングまで」のいずれかを関係単位で設定
- **入力**: 関係 ID、スコープ種別、適用開始日
- **処理**: 関係に紐づくデフォルトスコープを保持。イベント単位で上書き可能（F-024）。
- **出力**: スコープ台帳、変更履歴
- **受け入れ基準**:
  - スコープ変更は変更履歴 + 監査ログに残る
  - 「マエカクのみ」「商談のみ」「クロージングのみ」「契約後フォロー」は選択肢に存在しない
- **関連ロール**: `wholesaler_admin`
- **関連業務要件 ID**: docs/01 §3.3、提案書 §2.3

### 2.2 マスタ管理

#### F-011 場所提供元マスタ管理

- **目的**: docs/01 §4.1 / 提案書 §6.1。場所提供元の固定情報（連絡先・住所・契約形態）を再利用する
- **入力**: 名称、担当者、電話、メール、住所、エリア、契約形態（固定費型 / 成果報酬型 / その他）、固定費金額、成果報酬率、備考
- **処理**: マスタ登録。場所提供元対応 (F-017) からの参照キーになる。
- **出力**: マスタ一覧、詳細、変更履歴
- **受け入れ基準**:
  - 名称・住所必須
  - 二次店ロールはマスタ閲覧不可（API/UI 双方）
- **関連ロール**: `wholesaler_admin` / `wholesaler_event_team`
- **関連業務要件 ID**: docs/01 §4.1、提案書 F-001/F-024

#### F-012 商品・価格マスタ管理

- **目的**: docs/01 §7.3 — 商品マスタの価格改定が過去契約の粗利を変えないように、適用期間で履歴管理

#### F-064 レーンイベント開始/終了報告・成果報告の永続化

- **目的**: docs/01 §4.1（イベント当日報告）/ §7.8（月次集計の元データ）。レーンイベント (F-059) の各開催日について、運営者が Google スプレッドシート「レーンイベント」シートで運用していた **⑧開始/終了報告（報告者・開始/終了時間・特記事項）** と **⑨成果報告（抽選数・着座数・アポ数の両面/片面内訳、種別＝住宅設備/不動産）** を **実データとして永続化**する。現状はデモ生成（`apps/web/components/reports/result-report-data.ts` の `EventBasicReport` / `ResultReportData`）で未永続のため、当該デモと同一の項目定義のまま永続層へ移行することが本機能の主眼（差分は「未永続 → 永続」のみ。項目定義はデモと一致）。本機能は F-028〜F-030（単発イベント報告）の **レーンイベント版**であり、F-059 `LineEvent` の開催日を報告単位とする。
- **入力**:
  - **開始/終了報告**: 対象レーン (`LineEvent`) + 開催日、報告者、開始報告時刻 / 終了報告時刻、特記事項
  - **成果報告**: 対象レーン + 開催日、種別（住宅設備 / 不動産）、抽選数（両面 / 片面）、着座数（両面 / 片面）、アポ数（両面 / 片面）、コメント
- **処理**:
  - レーン × 開催日を報告キーとして報告レコードを永続化（デモの `EventBasicReport`＝開始/終了報告、`ResultReportData`＝成果報告に対応）。永続後はデモ生成を廃し DB を source of truth とする。
  - 両面 / 片面はそれぞれ独立した数値（同一指標の内訳）として保持し、合計＝両面＋片面を表示時に算出する。
  - 全 API クエリは `withTenant(ctx, ...)` を通し RLS + Prisma extension の二重防御を破らない（CLAUDE.md ルール 1）。報告は親 `LineEvent.wholesalerId` 経由で分離する。
  - 成果報告の数値は F-048 月次集計の元データになる（レーンイベント分の声かけ→アポ数系列を区分集計）。
  - 種別（住宅設備 / 不動産）・両面/片面ラベル・特記事項見出し等の UI 文言は `apps/web/lib/i18n/labels.ts` に集約（CLAUDE.md ルール 2）。
  - 本報告は仕入値・原価を一切扱わない（CLAUDE.md ルール 5、該当フィールドが DTO に存在しない）。
- **出力**: レーン × 開催日ごとの開始/終了報告レコード・成果報告レコード、レーン詳細 (F-059) の開催日別報告表示、月次集計 (F-048) の元データ
- **受け入れ基準**:
  - レーン × 開催日単位で開始/終了報告（報告者・開始時刻・終了時刻・特記事項）が保存・閲覧できる
  - 成果報告で抽選数・着座数・アポ数がそれぞれ両面 / 片面の内訳として保存され、合計が両面＋片面で表示される
  - 成果報告に種別（住宅設備 / 不動産）が保存される
  - 数値（抽選数・着座数・アポ数の両面/片面）は 0 以上の整数のみ受け付ける
  - 着座数 ≤ 抽選数、アポ数 ≤ 着座数のバリデーション（両面・片面それぞれで判定）
  - 終了報告時刻が開始報告時刻より前の場合は警告（後続報告は救済として可能、F-029 の運用方針と整合）
  - 永続後はデモ生成（`result-report-data.ts`）に依存せず DB の実データが表示される
  - 二次店ロールは自社が担当 / 共同開催に参加するレーンの報告のみ閲覧でき、他社二次店の報告は RLS + クエリ層で隔離される
  - 全クエリが `withTenant(ctx, ...)` 経由で RLS + Prisma extension の二重防御を通る
- **関連ロール**: `wholesaler_field_staff` / `wholesaler_event_team` / `dealer_admin` / `dealer_staff`（報告）、`wholesaler_admin`（閲覧・集計）
- **関連業務要件 ID**: docs/01 §4.1 / §7.8、提案書 F-008/F-105、F-028〜F-030 / F-048 / F-059

#### F-061 顧客詳細 案件情報 統合ビュー

- **目的**: docs/01 §4.1 / §1（Excel 散在の解消）。卸業者がイベント等で獲得した顧客について、契約 → ローン・入金 → 工事 → 認定・売電開始までの「案件情報（約 90 項目）」を **顧客詳細ページ 1 画面でカテゴリ別に一元閲覧** できるようにする。本機能は **既存集約を横断する集約（読み取り）ビュー**であり、新規トップレベル概念・並行集約は作らない（CLAUDE.md ルール 3、docs/05 §16.0 / §16.1 を尊重）。データの正は docs/05 §16.0 責務マトリクス（`Customer` = 個人属性・連絡先・担当 / `Deal` = 商談・提案金額 / `Contract` = 契約・契約日・運用ステータス・設備 ID / `ContractItem`・`GrossProfit` = 価格スナップショット・粗利 / `Construction` = 施工・着工・完工・売電 / `Application` = 認定・補助金申請 / `ContractPayment` = ローン・入金 / `ContractEquipment` = 設備スペック明細 / `CustomerActivity` = 商談ログ）に従う。
- **入力**: 顧客 ID（URL パラメータ）。表示フィルタとして契約（増設・買い替え時に 1 顧客 N 契約となりうる、docs/05 §16.8-1）の選択タブ。
- **処理**:
  - 顧客 ID から `Customer` を起点に、関連する `Deal` / `Contract` / `ContractItem` / `GrossProfit` / `Construction` / `Application` / `ContractPayment` / `ContractEquipment` / `CustomerActivity` を結合して取得（`withTenant(ctx, ...)` 経由、RLS + Prisma extension の二重防御を通す）。
  - 年齢は `Customer.birthDate` から表示時に算出（DB に冗長保持しない）。
  - 設備明細は `ContractEquipment.category`（PV / BT / EQ / IH / AC / ACCESSORY / GIFT）で判別し、カテゴリ別グループにまとめる。長尾属性（型番②・住設安心サポート・発電量保証・オプション付帯・三菱鍋型番・パワコン設置場所(新設/交換)）は `attributes Json` から展開して表示（docs/05 §16.3 のキー定義に従う）。
  - 仕入関連（仕入値 / 仕入合計 / その他原価 / 施工費の内訳）は **二次店ロールには DTO 層で物理除去**（CLAUDE.md #5）。本ビューが表示する金額は二次店に対し「契約金額 / 提案金額 / インセンティブ対象粗利 / インセンティブ額」に限定（docs/02 §7 Assumption 11）。
  - PII（氏名・電話・住所・生年月日）はマスキングポリシー（CLAUDE.md #6、docs/05 §16.4 / §6.5 `MaskingService`）を本一覧表示にも適用。既定: 氏名は姓のみ / 電話下 4 桁 / 住所は市区町村まで / 生年月日は年代のみ。`wholesaler_admin` は卸業者内の自顧客を非マスクで閲覧可、非マスク閲覧（`REVEAL_PII`）は監査ログ（F-055）に記録。`saas_admin` は常にマスク表示（Assumption 10）。
- **出力**: 顧客詳細ページの「案件情報」セクション。以下のカテゴリ別グルーピングで全項目を閲覧表示する（編集は F-062）:
  1. **基本情報**: お客様名 / フリガナ / 生年月日 / 年齢（算出） / 郵便番号 / 都道府県 / 市区町村 / 番地 / 電話番号 / メールアドレス / 築年日
  2. **体制**: アポ獲得者 / 営業担当者 / トス部署 / 所属部署
  3. **契約・金額**: ご契約日 / 契約書一式 URL / ご提案金額（税込） / 支払い回数 / 支払いステータス / 入金日 / 代理店支払日
  4. **ローン・団信**: ローン審査コール日時 / ローン会社 / 頭金 / 団体信用生命保険-契約有無 / ローン-備考 / 架電ステータス
  5. **工事・完工**: 現地調査日時(候補) / 工事日時(候補) / 現調日時 / 着工日 / 完工日 / サンキューコール日時 / 完工ステータス / 完工後ステータス / 不備解消ステータス / 不備内容
  6. **認定・設備**: 認定申請ステータス / 設備 ID / 売電開始日
  7. **設備明細**（カテゴリ別カード）: PV（契約有無 / 総合保証制度・発電量保証 / PV 延長保証制度 / メーカー / 型番 / 容量 / 枚数 / パワコン設置場所(新設) / オプション付帯） / BT（契約有無 / 設置場所 / 自然災害保証(有償) / 延長保証 / メーカー / 型番 / 容量） / EQ（契約有無 / 延長保証 / 導入状況 / 型番） / IH（契約有無 / 導入状況 / 型番） / AC（契約有無 / 住設安心サポート / 契約個数 / 型番①②） / 付帯商材（契約有無 / 契約個数 / 契約詳細 / 型番①② / パワコン設置場所(交換)） / プレゼント品（契約有無 / 個数 / 詳細 / 三菱鍋-型番）
  8. **商談ログ**: 既存 `CustomerActivity`（商談履歴）を時系列表示（F-038 / docs/05 §16.7）
  9. **備考**
- **受け入れ基準**:
  - 顧客詳細ページで上記 9 カテゴリの全項目が、対応データが存在する範囲で閲覧できる（未入力項目は「未設定」表示で欠落させない）
  - 年齢が生年月日から算出されて表示される（生年月日未登録時は年齢非表示）
  - 二次店ロールのレスポンス JSON に仕入値・仕入合計・その他原価の内訳フィールドが **存在しない**（`Object.keys` に出ない）
  - 二次店ロールは自社関与の顧客のみ本ビューに到達でき、他社二次店登録顧客は 403（F-032 の権限フィルタと整合）
  - 既定マスキング下で氏名は姓のみ・電話下 4 桁・住所市区町村まで・生年月日は年代のみで表示される。`wholesaler_admin` の非マスク表示操作が監査ログに残る
  - 設備明細はカテゴリ（PV/BT/EQ/IH/AC/付帯/プレゼント）ごとにグルーピングされて表示される
  - 契約・契約日・契約金額・施工日・申請ステータスは既存集約（`Contract`/`Construction`/`Application`）の値を表示し、本ビュー独自に二重保持しない（集約ビューである旨が満たされる）
- **関連ロール**: 全営業ロール（`wholesaler_admin` / `wholesaler_direct_sales` / `wholesaler_event_team` / `wholesaler_call_team` / `wholesaler_field_staff` / `dealer_admin` / `dealer_staff`、いずれも権限・PII フィルタ適用）
- **関連業務要件 ID**: docs/01 §4.1 / §9.4、提案書 F-009/F-107、docs/05 §16.0〜§16.7（特に §16.5 トレーサビリティ：S-113 顧客詳細）

#### F-062 顧客詳細 案件情報 編集（拡張要件）

- **目的**: F-061 で閲覧する案件情報のうち、既存集約に無い運用情報を顧客詳細ページから編集できるようにする。**閲覧（F-061）を MVP 必須要件とし、編集（CRUD）は段階実装（docs/05 §16.6 の 16-B〜16-D 単位）とする拡張要件**。
- **入力**: 編集対象に応じて以下を各 source of truth へ書き込む:
  - `Customer` 個人属性・連絡先・担当（F-031 の編集経路を流用）
  - `Contract` 付随運用情報（設備 ID / ローン審査コール日時 / サンキューコール日時 / 架電ステータス / 不備解消ステータス / 不備内容 / 完工後ステータス / 契約書一式 URL）
  - `Construction` 追加列（現地調査日時候補 / 工事日時候補 / 着工日 / 完工日 / 売電開始日 / 対応事業者名）
  - `Application` 認定申請ステータス（F-045 経路を流用）
  - `ContractPayment`（支払い回数 / 支払いステータス / ローン会社 / 頭金 / 団信契約有無 / ローン備考 / 入金日 / 代理店支払日）
  - `ContractEquipment`（カテゴリ別の設備スペック明細）
- **処理**:
  - 各項目は docs/05 §16.0 責務マトリクスが定める source of truth テーブルへ書き込み、F-061 ビューは更新後に再集計して表示する。
  - 金額・粗利は本機能では編集せず、契約金額・粗利は F-040〜F-042 の既存経路を正とする（CLAUDE.md #4 / #5）。
  - 新規子テーブル（`ContractEquipment` / `ContractPayment`）は親 `Contract.wholesalerId` 経由の相関 EXISTS で分離（docs/05 §16.4）。
  - 主要な編集操作は監査ログ（F-055）に記録。
- **出力**: 更新されたレコード、F-061 ビューへの即時反映
- **受け入れ基準**:
  - 編集が source of truth テーブル（`Customer` / `Contract` / `Construction` / `Application` / `ContractPayment` / `ContractEquipment`）にのみ書き込まれ、二重管理列を生成しない
  - 二次店ロールは仕入値・原価項目を入力・閲覧できない
  - 設備明細をカテゴリ別に追加・編集・削除でき、契約有無・型番・容量・各種保証が保存される
  - 金額・粗利は本画面から変更できない（既存 F-040〜F-042 経路へ誘導）
- **関連ロール**: `wholesaler_admin` / `wholesaler_direct_sales`（設備明細・施工・認定は権限により `wholesaler_event_team` も補助可）
- **関連業務要件 ID**: docs/01 §4.1、docs/05 §16.3 / §16.6（16-B〜16-D）/ §16.8

#### F-063 アポ取り顧客 住環境・家族属性ヒアリング管理

- **目的**: docs/01 §4.1（顧客・アポ管理）/ §9.4（情報開示制御）。レーンイベント等の催事でトスアップされたアポ取り顧客について、現場ヒアリングで取得した **住環境（既設設備の現況）・家族属性（年齢構成）・提案商材** を構造化して管理し、マエカク (F-035) と商談 (F-038) のクロスセル判断材料にする。運営者が Google スプレッドシート「レーンイベント」シートの「アポ取り顧客情報」（19 項目）で運用していた管理項目のうち、既存モデル（`Customer`）で未保持または非構造のものを永続化する。本機能は **F-031 顧客登録・編集の入力範囲拡張**であり、新規トップレベル概念は作らず `Customer` を source of truth とする（CLAUDE.md ルール 3、docs/05 §16.0 責務マトリクスに整合）。
- **既設設備（現況）と契約後設備（`ContractEquipment`）の概念分離**:
  - 本機能が扱う「既設設備」は **ヒアリング時点の顧客宅の現況**（ガス給湯機が付いている / エコキュートが設置済み / 太陽光が乗っている）を表し、**提案前のクロスセル判定材料**である。
  - 既存の `ContractEquipment`（PV/BT/EQ/IH/AC 等）は **契約成立後に当社が販売・施工する設備のスペック明細**（契約有無・型番・保証・価格スナップショット連動）であり、目的・ライフサイクル・source of truth が異なる。
  - 両者は **別概念として別テーブル（既設設備は `Customer` 子の現況ヒアリング、契約後設備は `Contract` 子）で管理**し、データを混在・相互参照させない。F-061 統合ビューでも「既設（現況）」と「契約設備」を別カテゴリとして区別表示する。
- **業務ルール（クロスセル判定材料）**:
  - **既設ガス給湯機あり → エコキュート（給湯機）提案**の候補。`gasWaterHeaterInstalled = true` をマエカク・商談時のフラグに用いる。
  - **既設エコキュートあり（設置日・メーカー保持）→ 蓄電池 / 太陽光のクロスセル**、または更新時期（設置日からの経年）に基づく買い替え提案の候補。
  - **既設太陽光あり（設置日・メーカー・容量・枚数保持）→ 蓄電池提案 / 増設提案**の候補。既設容量・枚数は増設可否の判断材料。
  - これらは判定**材料**としてのデータ保持であり、提案ロジックの自動化は MVP 範囲外（人間が判断する。docs/01 §4.3）。
- **入力**（F-031 顧客登録・編集フォーム内のヒアリングセクションとして追加。既存項目は再掲せず差分のみ列挙）:
  - **既存対応済み（再利用、再定義含む）**: トスアップ部署（`Customer.tossDept`）/ 営業担当者（`tossUp`/`closing` 担当）/ お客様名・フリガナ（`name`+`kana`）/ 郵便番号 / 都道府県 / 市区町村・番地（`prefecture`+`city`+`addressLine`）/ 家族構成（`household`）
  - **新規入力項目**:
    - ご提案商材（`proposedProduct`、トスアップ時に提案していた商材の自由記述 + 任意で商品マスタ参照）
    - ご主人年齢 / 奥様年齢 / お子様年齢（各任意。家族構成補助。年齢は **入力値を保持**＝ヒアリング時点のスナップショットであり、`Customer.birthDate` から算出する顧客本人年齢（F-061）とは別概念）
    - 案内者（`guideAttendee`、列挙：ご主人のみ / 奥様のみ / ご夫婦お揃い / その他。**`faceToFace`（主権者同席フラグ）とは意味が異なるため別項目として定義**。なお `faceToFace` は現状 DB（Prisma `Customer` モデル）には存在せず、デモ層（`apps/web/components/reports/customer-report-data.ts` / `labels.ts` / `customer-report-dialog.tsx`）にのみ存在する項目であり、永続化時に新 `guideAttendee`（案内者）とは別概念として保持／別列化する）
    - ガス給湯機設置有無（`gasWaterHeaterInstalled`、3 値：あり / なし / 不明）
    - エコキュート（設置有無 / 設置日 / メーカー）
    - 太陽光既設（設置有無 / 設置日 / メーカー / 容量[kW] / 枚数）
  - **構造化・分離が必要な既存項目**:
    - 電話番号は **固定電話 / 携帯電話の 2 系統に分離**（現行 `Customer.phone` 単一 → `landlinePhone` / `mobilePhone`。F-031 の重複チェックは両系統を対象とする）
    - マエカク電話希望日時（現行 `maekakuStatus` の有無のみ → `maekakuPreferredAt` として希望日時を永続化。F-035 マエカクキューの並びに利用）
    - アポイント取得日（現行 `Appointment.createdAt` 近似 → `Appointment.acquiredAt` を明示項目として保持。F-033 アポ登録の必須/任意項目に追加）
- **処理**:
  - 入力値は docs/05 §16.0 責務マトリクスに従い source of truth へ書き込む（顧客本人属性・既設設備現況・家族属性・提案商材は `Customer`、アポ取得日は `Appointment`）。
  - 全 API クエリは `withTenant(ctx, ...)` を通し、RLS + Prisma extension の二重防御を破らない（CLAUDE.md ルール 1）。既設設備現況は `Customer.wholesalerId` で分離する。
  - 既設太陽光の容量[kW]・枚数は 0 以上の数値、設置日は未来日不可（過去日 or 当日）。
  - 既設設備の現況フラグ（ガス給湯機 / エコキュート / 太陽光）は商談 (F-038) / マエカク (F-035) 画面で参照表示し、クロスセル候補のバッジを出す（自動提案はしない）。
  - 仕入値・原価項目は本機能では一切扱わない。二次店ロールに仕入関連を返さない（CLAUDE.md ルール 5、本機能は仕入値を持たないため該当フィールドが DTO に存在しない）。
  - UI 文言・選択肢ラベル（案内者・ガス給湯機 3 値・設置有無等）はすべて `apps/web/lib/i18n/labels.ts` に集約（CLAUDE.md ルール 2、コンポーネント内ハードコード禁止）。
- **出力**: 顧客レコードのヒアリング項目（既設設備現況・家族年齢・案内者・提案商材・分離電話・マエカク希望日時）、F-061 統合ビューの「ヒアリング（住環境・家族）」カテゴリ表示、F-038 / F-035 のクロスセル候補バッジ
- **個人情報マスキング（CLAUDE.md ルール 6 との関係）**:
  - **家族構成・ご主人/奥様/お子様年齢は個人情報（PII）として扱う**。既定マスキング下では年齢は具体値を伏せ「年代」または「未開示」で表示し、家族構成は概要（人数のみ等）に丸める。固定/携帯電話は氏名・住所と同様に下 4 桁マスクの対象。
  - **既設設備の現況（ガス給湯機 / エコキュート有無・設置日・メーカー / 太陽光有無・容量・枚数）は機微度が低い住環境情報**として、既定では非マスク表示を許容する（住所と紐づけて個人特定に至らない範囲）。ただし設置日・メーカーは住所等と組み合わせると特定性が上がるため、二次店ロールには「有無＋大分類」までとし詳細（設置日・メーカー・容量・枚数）は卸業者ロールに限定する（Assumption 11 / §9.4 と整合）。
  - 非マスク表示（`REVEAL_PII`）は監査ログ（F-055）に記録。`saas_admin` は常にマスク表示（Assumption 10）。
  - pino redact + Sentry beforeSend のフィールドリストに新規 PII 項目（家族年齢・分離電話）を追加し、監査ログ・エラー報告にもマスキングを適用（docs/05 §6.5 `MaskingService`）。
- **二次店表示可否**:
  - 二次店ロールは **自社関与の顧客のみ**本ヒアリング項目に到達可（F-032 の権限フィルタと整合）。他社二次店登録顧客は 403。
  - 二次店には家族年齢・分離電話は既定マスキングで表示。既設設備は「有無＋大分類」まで（設置日・メーカー・容量・枚数は不可視）。仕入値・原価はそもそも本機能に存在しない。
- **受け入れ基準**:
  - F-031 顧客フォームに「住環境・家族ヒアリング」セクションが追加され、提案商材・ご主人/奥様/お子様年齢・案内者・ガス給湯機有無・エコキュート（有無/設置日/メーカー）・太陽光既設（有無/設置日/メーカー/容量/枚数）が入力・保存できる
  - 電話番号が固定電話 / 携帯電話の 2 系統に分離して保存され、いずれも重複チェック対象になる
  - マエカク電話希望日時・アポイント取得日が明示項目として保存される
  - 案内者が「ご主人のみ / 奥様のみ / ご夫婦お揃い / その他」から選択でき、主権者同席フラグ（`faceToFace`。現状はデモ層のみに存在し DB 未永続。永続化時に別列化）とは別概念・別列として保存される
  - 既設太陽光の容量・枚数は 0 以上、設置日は未来日を拒否する
  - 既設設備の現況（ガス給湯機 / エコキュート / 太陽光）と契約後設備（`ContractEquipment`）が **別カテゴリ・別テーブル**で管理され、F-061 統合ビューで区別表示される
  - 既定マスキング下で家族年齢が年代/未開示、分離電話が下 4 桁で表示される。`wholesaler_admin` の非マスク表示操作が監査ログに残る
  - 二次店ロールには既設設備の設置日・メーカー・容量・枚数が返らない（DTO に存在しない／`Object.keys` に出ない）
  - 全クエリが `withTenant(ctx, ...)` 経由で RLS + Prisma extension の二重防御を通る
  - 既設ガス給湯機あり / 既設太陽光ありの顧客は、F-035 マエカク・F-038 商談画面でクロスセル候補バッジが表示される
- **関連ロール**: 全営業ロール（`wholesaler_admin` / `wholesaler_direct_sales` / `wholesaler_event_team` / `wholesaler_call_team` / `wholesaler_field_staff` / `dealer_admin` / `dealer_staff`、いずれも権限・PII フィルタ適用）
- **関連業務要件 ID**: docs/01 §4.1 / §9.4、提案書 F-009/F-106/F-204、docs/05 §16.0、F-031 / F-035 / F-038 / F-061

### 2.6 商談・契約・粗利

#### F-038 商談・クロージング管理

- **目的**: docs/01 §3.3 / 提案書 §6.12
- **入力**: 顧客、担当種別（卸業者 / 二次店）、担当者、担当二次店、初回訪問日時、ステータス、提案商品、提案金額、粗利見込み、契約予定日、失注理由、次回アクション、備考
- **処理**:
  - 商談スコープに応じてアクセス制御（イベント単位上書き F-024 を優先）
  - ステータス遷移（初回訪問予定 → 訪問済み → 提案中 → 見積提出 → 検討中 → 契約見込み → 契約 / 失注）
- **出力**: 商談レコード、通知
- **受け入れ基準**:
  - 「アポ獲得まで」スコープの二次店は商談ステータス変更不可（閲覧のみ）
  - 契約ステータス遷移時、F-040 契約登録画面へ遷移誘導
- **関連ロール**: `wholesaler_direct_sales` / `dealer_admin` / `dealer_staff`（スコープ次第）
- **関連業務要件 ID**: docs/01 §3.3 / §7.6、提案書 F-013/F-107/F-109、§10.9

#### F-039 二次店商談・クロージング報告確認

- **目的**: 提案書 F-014
- **入力**: フィルタ（対象月・二次店・ステータス）
- **処理**: 二次店からの商談更新を卸業者側でレビュー
- **出力**: 一覧、通知
- **受け入れ基準**: 二次店が契約ステータスに上げた時、卸業者側通知に乗る
- **関連ロール**: `wholesaler_admin` / `wholesaler_direct_sales`
- **関連業務要件 ID**: 提案書 F-014、§9.1

#### F-040 契約登録

- **目的**: docs/01 §7.3 / 提案書 §6.13
- **入力**: 商談、顧客、契約日、契約金額、販売商品（明細は F-041）、メーカー、パネル容量、蓄電池有無、補助金利用有無、契約書ファイル
- **処理**:
  - 契約成立日 = 契約日。`cancel_deadline = 契約日 + F-015 期限日数`
  - インセンティブ率・スコープを契約に **スナップショット**
  - 自社開催由来案件は二次店インセンティブ対象外フラグを立てる
  - 共同開催由来は手動調整待ち (F-047) を起票
- **出力**: 契約レコード（ID、ステータス）、関連明細・粗利・インセンティブの空レコード
- **受け入れ基準**:
  - 契約日・契約金額・販売商品 1 件以上必須
  - 契約成立時にインセンティブが「確定（支払予定額）」となる（F-046）
  - キャンセル期限は契約レコードに保持されており、F-015 設定変更で過去契約は変わらない
- **関連ロール**: `wholesaler_admin` / `wholesaler_direct_sales` / 権限付き二次店
- **関連業務要件 ID**: docs/01 §7.2 / §7.3、提案書 F-015、§6.13

#### F-041 契約明細登録（価格スナップショット）

- **目的**: docs/01 §7.3 — 契約明細スナップショット
- **入力**: 契約、商品（F-012 から選択）、数量、契約時点の仕入値、契約時点の二次店卸値、契約時点の販売価格、単位
- **処理**:
  - 商品マスタから契約日時点の有効レコード（適用期間内）を引いて **明細にコピー**
  - 商品マスタの後続改定は明細に反映しない
  - 明細合計を契約に集計（仕入値合計、二次店卸値合計、販売価格合計）
- **出力**: 明細レコード、契約集計値
- **受け入れ基準**:
  - 契約日時点で適用中でない商品マスタは選択不可
  - 明細 1 件以上ない契約は粗利計算 (F-042) に進めない
  - 明細登録後、商品マスタを改定しても契約明細の値は不変
- **関連ロール**: `wholesaler_admin` / `wholesaler_direct_sales`
- **関連業務要件 ID**: docs/01 §7.3、提案書 F-017、§10.11

#### F-042 粗利計算（自動・手動調整）

- **目的**: docs/01 §7.3 / 提案書 §6.16
- **入力**: 実販売価格、施工費、その他原価、値引き額、インセンティブ対象粗利種別（案件粗利 / 卸粗利 / 手動指定）、手動指定額（手動時）
- **処理**:
  - 案件粗利 = 実販売価格 − 商品仕入値合計 − 施工費 − その他原価 − 値引き額
  - 卸粗利 = 二次店卸値合計 − 商品仕入値合計
  - 粗利率 = 案件粗利 / 実販売価格
  - インセンティブ対象粗利 = 種別に応じて上記いずれかを使用、手動指定時は入力額
- **出力**: 粗利レコード、契約に紐づく集計値
- **受け入れ基準**:
  - 実販売価格必須、未入力時は粗利未計算ステータス
  - インセンティブ対象粗利の手動調整は監査ログ + 変更履歴
  - 粗利が 0 以下の場合インセンティブ計算 (F-046) は 0 円
- **関連ロール**: `wholesaler_admin` / `wholesaler_direct_sales`
- **関連業務要件 ID**: docs/01 §7.3、提案書 F-018、§10.12

#### F-043 契約キャンセル処理

- **目的**: docs/01 §7.2 — 期限内取消し / 期限後負調整
- **入力**: 契約、キャンセル日、キャンセル理由
- **処理**:
  - キャンセル日 ≤ `cancel_deadline` の場合: インセンティブ確定額を取消し（インセンティブ対象外フラグを立てる）
  - キャンセル日 > `cancel_deadline` の場合: 当該インセンティブを翌月以降の支払額から減算する **負調整レコード** を作成
  - 契約ステータスを「キャンセル」に変更、施工・申請・月次集計に反映
- **出力**: キャンセルレコード、負調整レコード（期限後のみ）、通知
- **受け入れ基準**:
  - 期限内キャンセル後、対応する月次の確定済みインセンティブが取消し済みであることを月次画面で確認できる
  - 期限後キャンセルの負調整は次回月次インセンティブ計算 (F-046) に自動で適用される
  - キャンセル操作は監査ログに残る
- **関連ロール**: `wholesaler_admin`
- **関連業務要件 ID**: docs/01 §7.2、Open Q3

#### F-044 施工状況管理

- **目的**: 提案書 §6.18 — ステータス追跡のみ
- **入力**: 契約、施工業者、現地調査日、施工予定日、施工完了日、ステータス、施工費用、添付ファイル
- **処理**: ステータス遷移（依頼前 → 依頼済み → 現地調査済み → 施工中 → 完了 / 中断）
- **出力**: 施工レコード
- **受け入れ基準**:
  - 施工予定日 7 日前に担当者へ通知 (F-052)
  - 施工費用更新時、粗利 (F-042) が自動再計算される
- **関連ロール**: `wholesaler_admin` / `wholesaler_direct_sales`
- **関連業務要件 ID**: 提案書 F-019、§9.1

#### F-045 補助金等申請管理

- **目的**: 提案書 §6.19 — ステータス追跡のみ
- **入力**: 契約、申請種別、申請先、申請予定日、申請日、承認日、ステータス、補助金見込み額、補助金確定額、添付ファイル
- **処理**: ステータス遷移
- **出力**: 申請レコード
- **受け入れ基準**: 申請期限 14 日前に担当者へ通知 (F-052)
- **関連ロール**: `wholesaler_admin` / `wholesaler_direct_sales`
- **関連業務要件 ID**: 提案書 F-020、§9.1

### 2.7 インセンティブ・月次クローズ

#### F-046 インセンティブ自動計算

- **目的**: docs/01 §7.2 — 契約成立時に確定 / 提案書 §6.17
- **入力**: 契約、粗利 (F-042)、関係単位のインセンティブ率スナップショット (F-040)
- **処理**:
  - 自社開催のみ案件、キャンセル案件、粗利 ≤ 0 はインセンティブ 0 円
  - 共同開催案件は手動調整待ちフラグ + 標準計算値（下書き）を提示 (F-047)
  - インセンティブ額 = インセンティブ対象粗利 × インセンティブ率
  - 契約成立日属する暦月の「確定インセンティブ」として記録
- **出力**: インセンティブレコード（確定額 / 下書き）、通知
- **受け入れ基準**:
  - 契約成立直後にインセンティブ確定額が表示される
  - 後続のキャンセル (F-043) が反映される
  - インセンティブ率未設定の関係で契約を成立させると警告 + 計算 0 円
- **関連ロール**: `wholesaler_admin`
- **関連業務要件 ID**: docs/01 §7.2、提案書 F-022、§10.13

#### F-047 共同開催インセンティブ手動調整

- **目的**: docs/01 §7.7 — 共同開催は案件ごと手動調整、MVP は調整 UI + 監査ログ
- **入力**: 契約、調整後の各二次店インセンティブ額、調整理由
- **処理**:
  - 下書き提示は F-046 で自動計算した値
  - 確定操作で確定額を上書き、監査ログ + 変更履歴
- **出力**: 確定額、履歴
- **受け入れ基準**:
  - 手動調整しないと共同開催案件は月次確定 (F-050) に進めない
  - 監査ログから「誰がいつ何を変更したか」が追跡できる
- **関連ロール**: `wholesaler_admin`
- **関連業務要件 ID**: docs/01 §7.7

#### F-048 月次集計

- **目的**: docs/01 §7.8 / 提案書 §6.20
- **入力**: 対象年月、卸業者テナント、二次店フィルタ
- **処理**:
  - 区分: 自社開催 / 二次店開催 / 共同開催 / 全体
  - 集計項目（提案書 §6.20 の自動集計項目に準拠）: 実施イベント数、対応店舗数、稼働日数、声かけ数、アンケート取得数、アポ数、有効アポ数、マエカク通過数、初回訪問数、商談数、契約数、契約金額、粗利、インセンティブ見込み額
  - 暦月固定（年度開始月は表示順だけに影響、F-016）
  - 集計はオンデマンド + キャッシュ（月次クローズで確定スナップショット）
- **出力**: 集計レポート、二次店別 / 体制別ビュー
- **受け入れ基準**:
  - 全体 = 自社 + 二次店 + 共同 の合算が保証される
  - 二次店ロールは自社分のみ表示
- **関連ロール**: `wholesaler_admin` / `wholesaler_event_team` 他
- **関連業務要件 ID**: docs/01 §7.8、提案書 F-021/F-110

#### F-049 月次報告コメント入力・提出

- **目的**: 提案書 §6.20
- **入力**: 主な成果、課題、改善アクション、翌月重点店舗、翌月施策、二次店コメント
- **処理**: ステータス遷移（下書き → 提出済み → 確認済み → 確定）。二次店が提出 → 卸業者が確認 → 確定。
- **出力**: 月次報告レコード
- **受け入れ基準**:
  - 確定後はコメント編集不可
  - 提出時に卸業者宛通知
- **関連ロール**: `dealer_admin`（提出）、`wholesaler_admin`（確認・確定）
- **関連業務要件 ID**: 提案書 F-021/F-110

#### F-050 月次報告確定・ロック

- **目的**: docs/01 §10 リスク — 監査ログ上の手動補正件数を可視化
- **入力**: 対象年月、卸業者テナント
- **処理**:
  - 集計値スナップショットを確定テーブルに保存
  - 確定後は当該月の粗利・インセンティブ手動調整不可（緊急対応として `wholesaler_admin` のみアンロック可、監査ログ）
- **出力**: 確定レコード、確定後ビュー
- **受け入れ基準**:
  - 確定済み月の契約に手動調整を試みると 409
  - アンロックは監査ログに残る
- **関連ロール**: `wholesaler_admin`
- **関連業務要件 ID**: docs/01 §7.2 / §7.8

#### F-059 レーンイベント管理

- **目的**: docs/01 §4.1。懇意の場所提供元と月単位で複数の開催日を契約する「レーンイベント」を管理する。1 日 1 件で場所取りする単発イベント (F-018) と異なり、1 行が対象月内の複数開催日を保持する。
- **入力**: レーン名（必須）、場所提供元、エリア、対象月（YYYY-MM、必須）、開催日（対象月内の複数日、最低 1 日必須）、ステータス（確認中 / 確定 / 中止）、場所提供元との契約形態（固定費型 / 成果報酬型 / その他）、固定費（日当たり報酬額）、成果報酬率、メモ
- **処理**: レーン登録（status は確認中=DRAFT / 確定=CONFIRMED / 中止=CANCELLED）。一覧は対象月と場所提供元で絞り込み、月の各日に対する開催有無をドットグリッドで表示する。
- **出力**: レーン一覧（レーン名・場所提供元・開催日グリッド・エリア・開催回数・登録日時）、レーン詳細（基本情報・契約条件・開催日カレンダー・ステータス）
- **受け入れ基準**:
  - レーン名・対象月・開催日（1 日以上）は必須
  - 開催回数 = 開催日配列の件数
  - 単発イベントと同じく二次店ロールは閲覧不可（メニュー非表示、URL 直叩きは 403）
  - 固定費・成果報酬率は場所提供元との契約条件であり二次店には公開しない
- **関連ロール**: `wholesaler_admin` / `wholesaler_event_team`
- **関連業務要件 ID**: docs/01 §4.1、提案書 §6.2

#### F-060 二次店レーン希望（一覧確認）

- **目的**: docs/01 §4.1 / 提案書 §6.4。二次店が月単位で「希望するレーン（＝希望場所 × 希望開催日）」を**希望順位付きで複数件**提出する。本機能は **卸業者側の一覧確認ビュー**で、二次店ごとの希望提出をアコーディオン形式で一望し、開催体制決定 (F-023) / レーンイベント確定 (F-059) の材料にする。
  - **構造変更（旧 → 新）**: 旧仕様は「卸業者が先に作成した確定レーン (F-059 `LineEvent`) を二次店が優先順位付きで選ぶ」トップダウン構造だった。新仕様は運営者が確定した「二次店希望情報」管理項目（対象月 / 二次店名 / 希望レーン数 / 特記事項）に合わせ、**二次店が確定レーンに依存せず、希望場所と希望開催日を自由記述で複数件提出できるボトムアップ構造**に作り変える。希望明細は `LineEvent` への参照を必須としない（F-059 と疎結合）。
- **データ構造**: 「対象月 × 二次店」を一意キーとする提出ヘッダ `LanePreference` と、その配下の**希望レーン明細** `LanePreferenceItem`（従来名を再定義）で構成する。
  - `LanePreference`（提出ヘッダ）: `relationshipId`（提出二次店）、`targetMonth`（対象月 YYYY-MM）、`note`（特記事項＝従来 `comment` を維持・改称）、提出日時
  - `LanePreferenceItem`（希望レーン明細、1 提出に複数）:
    - `priority`（希望順位 = 希望①②③…。表示順を兼ねる。既存 `priority` 列を流用）
    - `venueLabel`（希望場所の表示用ラベル文字列。例「カインズ 大宮店」。**必須**・一次ソース）
    - `venueProviderId` / `storeId`（マスタ：場所提供元・店舗への**任意リンク**。マスタ未登録の場所も登録できるよう、リンクは任意でラベル文字列が一次ソース）
    - `lineEventId`（確定レーンへの**任意リンク**。旧仕様で必須だった参照を任意リンクに降格）
    - `desiredDates`（希望開催日。週単位で複数の希望日帯を保持。例 7/7~8・7/12~13・7/14~15。保持方式は Open Q26 で確定するが、要件上は「複数の日帯（週ごとの希望開催日）」を表現できること）
    - `memo`（任意のレーン別メモ）
  - **「希望レーン数」は冗長保持しない**＝`LanePreferenceItem` の件数から導出する。
- **入力（卸業者側フィルタ）**: 対象月（YYYY-MM、デフォルト当月）、二次店（プルダウン、ACTIVE な関係から選択）、検索 / クリア
- **入力（二次店側提出フォーム、F-021 と連動）**: 対象月、希望レーン明細の行追加 / 削除（希望順位の自動採番・並べ替え）、各行の希望場所ラベル（必須・マスタ補完サジェスト併用可）、希望開催日（週単位の日帯を複数追加）、レーン別メモ、特記事項。MVP のフォーム実装スコープは後述。
- **処理**: `LanePreference` を対象月・二次店で絞り込み、提出日時の降順で取得。各提出の `LanePreferenceItem` を `priority` 昇順（＝希望①→②→③…）に並べ、各明細の希望場所ラベル・希望開催日帯・任意リンク先（あれば確定レーン名 / 場所提供元名）を結合して表示する。希望レーン数 = 明細件数を見出しに表示する。
- **出力**: 二次店ごとのアコーディオン行（二次店名 + 提出日時 + 希望レーン数バッジ + 開閉）。展開時は希望順位カード（希望① / 希望② / 希望③…）を `priority` 昇順で横並び表示し、各カードに希望場所ラベル、希望開催日チップ群（7/7~8 等）、（任意リンクがあれば）確定レーン名・場所提供元名、レーン別メモを表示。最下部に特記事項 (`note`) を表示。CSV 出力はプレースホルダ（Phase 2）。
- **受け入れ基準**:
  - 同一二次店 × 同一対象月の提出は一意（`@@unique([relationshipId, targetMonth])`、再提出は更新扱い）
  - 希望レーン明細は `priority` 昇順で表示される（priority=1 が希望①）。表示順は希望順位昇順に固定
  - **希望場所ラベル (`venueLabel`) は必須**で、マスタ（場所提供元 / 店舗）未登録の場所でも提出・表示できる（ラベル文字列が一次ソース）
  - **希望明細は確定レーン (`LineEvent`) への参照を必須としない**（旧仕様との差異）。`lineEventId` が無い明細（＝二次店の自由記述希望）も正常に一覧表示される
  - 希望レーン数は明細件数から導出され、別カラムに冗長保持されない（件数とバッジ表示が常に一致する）
  - 各明細は複数の希望開催日帯（週ごと）を保持・表示できる（例 7/7~8・7/12~13・7/14~15 が 1 明細内に並ぶ）
  - 二次店ロールは**他社二次店の希望を閲覧不可**。卸業者側一覧は `relationship_id` / `wholesaler_id` で分離され、RLS + クエリ層で隔離される。卸業者側一覧メニューは二次店ロールに非表示（URL 直叩きは 403）
  - 仕入値・卸非公開項目（場所提供元との固定費 / 成果報酬率等）は本一覧に含めない（希望明細が保持するのは二次店由来の希望情報のみ）
- **MVP（Phase 1）スコープ**: 卸業者側の一覧確認ビューを中心に実装する。二次店側提出フォームは F-021（二次店希望回答、P0）の一機能として MVP で薄く実装するか Phase 2 とするかを Open Q24 で判断する。MVP で一覧側のみ先行する場合はシード / 手動投入データを表示し、新データ構造（ラベル必須・任意リンク・複数日帯）を前提に描画する。
- **F-059 との関係**: 希望提出は確定レーン (F-059 `LineEvent`) の存在に依存しない（ボトムアップ）。`lineEventId` の任意リンクは「二次店の希望がすでに確定済みレーンに対応する場合の突合」用であり、未リンクの希望から卸業者が新規レーンを起こす運用も成立する。
- **関連ロール**: `wholesaler_admin` / `wholesaler_event_team`（一覧確認）、`dealer_admin`（提出、F-021 連動）
- **関連業務要件 ID**: docs/01 §4.1（二次店希望収集）、提案書 §6.4 F-004 二次店希望店舗管理 / §6.2

## 3. ユースケース

各ユースケースは「アクター・前提・手順・結果」形式。スコープ・ロール・関連機能 ID を明記する。

### UC-01 場所提供元調整 → イベント候補 → 希望提出 → 開催体制決定 → 実施 → 報告

- **アクター**: `wholesaler_event_team`, `wholesaler_admin`, `dealer_admin`, `dealer_staff`, `wholesaler_field_staff`
- **前提**: 卸業者と二次店の関係 (F-009) が確立済み、商品マスタ (F-012) と場所提供元マスタ (F-011) が登録済み
- **手順**:
  1. `wholesaler_event_team` が場所提供元と LINE/電話で調整し、F-017 場所提供元対応で「条件確認中 → 確定」に更新
  2. F-018 でイベント候補を「下書き」として登録（対象月、実施予定日、店舗名、回答期限）
  3. ステータスを「希望受付中」に変更し、F-019 で対象二次店に公開（場所・日にち・回答期限のみ）
  4. `dealer_admin` が F-021 で月単位の希望店舗を複数選択して提出
  5. 期限到来後、`wholesaler_event_team` が F-022 で二次店別/店舗別の希望状況を確認
  6. F-023 で開催体制を決定（例: 共同開催、担当二次店 = A 社、必要人数 = 3）
  7. F-025 で自社要員シフトを割当（重複チェック・必要人数充足チェック）
  8. F-024 で当該イベントの二次店スコープを「商談・クロージングまで」に上書き
  9. 当日: `wholesaler_field_staff` と `dealer_staff` がそれぞれ F-028 / F-029 / F-030 で開始・終了・成果報告
  10. F-031 / F-033 でアポ顧客を登録
- **結果**: イベント実施が確定し、成果が月次集計 (F-048) に反映される
- **関連機能**: F-009〜F-035, F-048

### UC-02 アポ獲得 → マエカク → 結果連絡 → 商談・クロージング → 契約成立

- **アクター**: `dealer_staff` または `wholesaler_field_staff`（アポ獲得）、`wholesaler_call_team`、`wholesaler_direct_sales` または `dealer_admin`（商談）
- **前提**: 当該二次店のスコープ = 「商談・クロージングまで」（または「初回訪問まで」）
- **手順**:
  1. イベント当日に `dealer_staff` が F-031 顧客 + F-033 アポを登録
  2. アポステータスが「未確認」となり、`wholesaler_call_team` のマエカクキュー (F-034) に乗る
  3. F-035 でマエカク実施、コール結果「承認」を登録
  4. F-036 で結果を担当二次店へ連絡（注意事項・訪問日時確認）
  5. `dealer_admin` が F-037 でマエカク結果を確認
  6. F-038 で初回訪問〜商談ステータスを更新（「初回訪問済み」「提案中」「契約見込み」）
  7. ステータス「契約」に上げると F-040 契約登録画面へ自動誘導
  8. F-041 で契約明細を登録（商品マスタから契約日時点の価格を自動コピー）
  9. F-042 で粗利が自動計算 → F-046 でインセンティブが即時確定
- **結果**: 契約成立日に二次店のインセンティブ確定額が表示される
- **関連機能**: F-031, F-033, F-035〜F-042, F-046

### UC-03 契約成立 → 契約明細登録 → 粗利計算 → インセンティブ確定

- **アクター**: `wholesaler_direct_sales`, `wholesaler_admin`
- **前提**: F-038 商談ステータスが「契約見込み」
- **手順**:
  1. F-040 で契約基本情報を登録（契約日、契約金額、契約書ファイル）
  2. F-041 で複数明細を登録。商品マスタの「契約日時点で適用中」のレコードから仕入値・二次店卸値・希望販売価格を **明細にコピー**
  3. F-042 で実販売価格・施工費・その他原価・値引きを入力 → 案件粗利・粗利率・インセンティブ対象粗利が自動算出
  4. インセンティブ対象粗利種別を「案件粗利」「卸粗利」「手動指定」から選択
  5. F-046 が契約成立を検知し、関係単位のインセンティブ率を契約にスナップショット → 確定額を計算
  6. 商品マスタを後日改定しても、当該契約の粗利・インセンティブは不変であることを確認 (F-012 / F-041)
- **結果**: 契約明細スナップショットが保持され、過去契約の粗利が変動しない
- **関連機能**: F-012, F-040〜F-042, F-046

### UC-04 キャンセル発生（期限内取消し / 期限後負調整）

- **アクター**: `wholesaler_admin`
- **前提**: F-040 で契約成立済み、F-046 でインセンティブ確定済み
- **手順 A（期限内取消し）**:
  1. 契約日から 5 日後にキャンセル発生
  2. F-043 でキャンセル日・理由を入力 → `cancel_deadline`（契約日 + 8 日）以内のため、確定インセンティブを取消し
  3. 契約ステータスが「キャンセル」になり、月次集計 (F-048) からも除外される
- **手順 B（期限後負調整）**:
  1. 契約日から 15 日後にキャンセル発生
  2. F-043 でキャンセル日・理由を入力 → 期限超過のため、当該インセンティブ額を翌月以降の支払額から減算する **負調整レコード** を作成
  3. 翌月の F-046 計算時に負調整が自動適用される
  4. F-048 月次集計で「負調整内訳」が確認できる
- **結果**: docs/01 §7.2 のキャンセル処理が業務通りに反映される
- **関連機能**: F-015, F-040, F-043, F-046, F-048

### UC-05 月次クローズ（自動集計 → 確定 → 二次店向け確認）

- **アクター**: `wholesaler_admin`, `dealer_admin`
- **前提**: 対象月の契約・粗利・インセンティブが揃っている。共同開催案件は F-047 で手動調整済み。
- **手順**:
  1. 月末翌日に F-048 が暦月集計を自動更新（自社/二次店/共同/全体の 4 区分）
  2. `dealer_admin` が F-049 で月次報告コメント（成果・課題・翌月施策）を入力 → 「提出済み」
  3. `wholesaler_admin` が F-049 でコメントを確認 → 「確認済み」
  4. F-050 で月次確定を実行（集計値スナップショット保存、当該月の手動調整ロック）
  5. F-051 で二次店が自社分の成績・インセンティブを確認（仕入値非表示、卸業者ごとに独立表示）
  6. 確定後に発生したキャンセル等は F-043 経由で翌月以降に影響する
- **結果**: 月次クローズが完了し、当該月の数値が確定する
- **関連機能**: F-042, F-046〜F-051

### UC-06 顧客の案件情報を顧客詳細ページで一元閲覧する

- **アクター**: `wholesaler_direct_sales`, `wholesaler_admin`（閲覧）、`dealer_admin`（自社関与顧客の閲覧）
- **前提**: 対象顧客 (F-031) と、その契約 (F-040)・契約明細 (F-041)・粗利 (F-042)・施工 (F-044)・認定/補助金申請 (F-045)・ローン入金 (`ContractPayment`)・設備明細 (`ContractEquipment`) が登録されている（一部未登録でも可）
- **手順**:
  1. `wholesaler_direct_sales` が F-032 顧客一覧から対象顧客を開く（顧客詳細ページ S-113）
  2. F-061 が `Customer` を起点に関連集約を横断結合し、9 カテゴリ（基本情報 / 体制 / 契約・金額 / ローン・団信 / 工事・完工 / 認定・設備 / 設備明細 / 商談ログ / 備考）でグルーピング表示
  3. 年齢が生年月日から算出表示され、未入力項目は「未設定」と表示される
  4. 設備明細は PV / BT / EQ / IH / AC / 付帯 / プレゼントのカテゴリ別カードで表示
  5. `wholesaler_admin` が必要に応じ非マスク表示（`REVEAL_PII`）を実行 → 監査ログ (F-055) に記録
  6. （拡張）F-062 で運用情報（架電ステータス・着工日・入金日・設備明細等）を該当 source of truth テーブルへ編集 → ビューに即時反映
- **代替フロー（二次店アクター）**: `dealer_admin` が自社関与顧客を開くと、仕入値・原価内訳を含まないビューが返り、PII は既定マスキングで表示される。他社二次店登録顧客への到達は 403。
- **結果**: 契約〜工事〜売電開始までの案件情報が顧客詳細 1 画面で俯瞰でき、Excel 散在（docs/01 §1）が解消される。表示は既存集約の集約ビューであり値の二重管理は発生しない。
- **関連機能**: F-031, F-032, F-040〜F-045, F-055, F-061, F-062

### UC-07 レーンイベント当日報告（開始/終了・成果）と住環境ヒアリングのクロスセル活用

- **アクター**: `wholesaler_field_staff` または `dealer_staff`（当日報告・ヒアリング）、`wholesaler_call_team`（マエカク）、`wholesaler_direct_sales` または `dealer_admin`（商談）
- **前提**: レーンイベント (F-059) と開催日が登録済み、当該二次店のスコープが「初回訪問まで」以上
- **手順**:
  1. レーン開催当日、`wholesaler_field_staff` が F-064 で開始報告（報告者・開始時刻・特記事項）を入力 → 終了時に終了報告を入力
  2. 撤収後に F-064 で成果報告を入力（種別＝住宅設備、抽選数 両面/片面、着座数 両面/片面、アポ数 両面/片面）。着座 ≤ 抽選、アポ ≤ 着座のバリデーションが効く
  3. 獲得アポについて F-031 / F-063 で顧客を登録。住環境・家族ヒアリング（ご主人/奥様/お子様年齢・案内者＝ご夫婦お揃い・既設ガス給湯機あり・既設太陽光なし・提案商材）を入力。固定/携帯電話・アポ取得日・マエカク希望日時を保存
  4. `wholesaler_call_team` が F-035 マエカクで顧客を開くと、既設ガス給湯機ありのクロスセル候補バッジ（エコキュート提案）が表示される
  5. F-038 商談で `wholesaler_direct_sales` が既設設備現況を参照し、エコキュート提案を起票
  6. F-064 成果報告は F-048 月次集計のレーンイベント区分の元データになる
- **結果**: レーンイベントの開始/終了・成果が実データとして永続化され（デモ廃止）、ヒアリングした既設設備が商談のクロスセル判断材料として機能する
- **関連機能**: F-031, F-035, F-038, F-048, F-059, F-063, F-064

## 5. 非機能要件

### 5.1 性能

- 業務時間帯のレスポンス: 一覧画面 95%tile < 800ms、詳細 < 500ms
- 月次集計 (F-048): 1 卸業者あたり 100 二次店 / 1,000 契約規模で < 5 秒
- 一覧はページネーション（50 件/ページ）+ 必須インデックス（`wholesaler_id`, `relationship_id`, `created_at`, `target_month` 等の複合）
- 同時利用想定: 1 卸業者あたり 50 アクティブセッション、SaaS 全体 500 セッション

### 5.2 可用性

- **SLA: 業務時間帯 8:00–22:00 JST で 99.5%**
- 深夜帯 (22:00–8:00) のメンテナンスは事前告知の上で許容
- 復旧時間目標 (RTO): 4 時間（業務時間帯）
- データ復旧目標 (RPO): 24 時間（日次バックアップ）

### 5.3 セキュリティ

- **認証**: メール+パスワード必須、`saas_admin` / `wholesaler_admin` は 2FA (TOTP) 必須化 (F-002)
- **認可**: テナント分離キー（`wholesaler_id` / `relationship_id`）を全 API で強制
- **仕入値非表示**: 二次店ロールには仕入値フィールドを返さない（API レイヤで除去）
- **個人情報マスキング**: 監査ログ・運営者画面・メール本文・顧客詳細案件ビュー (F-061) では電話番号（固定/携帯とも下 4 桁のみ表示）・住所（市区町村まで）・氏名（姓のみ）・生年月日（年代のみ）・家族年齢（ご主人/奥様/お子様年齢は年代/未開示）・家族構成（概要に丸める）をマスク。既設設備の現況詳細（エコキュート/太陽光の設置日・メーカー・容量・枚数）は二次店ロールに不可視（卸業者ロール限定、F-063）。`wholesaler_admin` の非マスク表示 (`REVEAL_PII`) は監査ログに記録
- **保護対象データ**: 顧客個人情報、契約書ファイル、価格情報、インセンティブ額
- **暗号化**: パスワードはハッシュ化（bcrypt または argon2）、契約書ファイルは R2 で AES-256 保存、HTTPS 必須
- **セッション**: 24 時間有効、業務時間外ログインは監査ログに記録
- **CSRF / XSS**: フレームワーク標準対策 + CSP ヘッダ
- **失敗ロック**: ログイン 5 回失敗で 15 分ロック
- **退職者対応**: ユーザー停止 (F-006) で即時ログイン不可

### 5.4 監視・ログ

- **アプリログ**: 構造化 JSON ログ、リクエスト ID で横断トレース
- **エラー監視**: 5xx エラー / ジョブ失敗をリアルタイム通知
- **業務ログ**: 監査ログ (F-055) でビジネス操作を追跡
- **コスト観測**: メール送信数 / LINE 送信数 / ストレージ使用量を日次集計
- **稼働状況**: アップタイム監視（外形監視）
- **アラート閾値**: 5xx > 1%/分、ジョブキュー滞留 > 100 件、DB 接続失敗 > 3 回連続

### 5.5 拡張性

- **LINE Messaging API 連携 (Phase 2)**: 通知配信チャネルとしてプラグイン
- **CSV/Excel インポート (Phase 2)**: マスタ系 (顧客・二次店関係・商品・契約履歴) の取込
- **施工業者向け画面 (Phase 3)**: 施工業者は当面マスタ扱い、Phase 3 で簡易画面検討
- **BI 拡張 (Phase 2 以降)**: 集計テーブル基盤を予め分離
- **多通貨・多言語 (Phase 4 以降)**: 現状は JPY / 日本語のみ
- **マルチクラウド**: Railway を初期、将来的に AWS / GCP への移行余地を残す（Prisma + Postgres + R2 互換）

### 5.6 ユーザビリティ・アクセシビリティ

- **レスポンシブ Web**: 主要画面はモバイル (375px) / タブレット / PC で最適化
- **スマホ最優先**: 二次店担当・現場要員のフロー（シフト確認・報告・アポ登録）
- **キーボードショートカット**: 卸業者本部の一覧画面に絞込みショートカット
- **WCAG 2.1 AA 準拠**: コントラスト、フォーカス表示、aria-label

### 5.7 データ保持・運用

- **監査ログ**: 3 年保持（Open Q10 で再確認）
- **顧客個人情報**: テナント停止後 90 日でハードデリート（要法務確認）
- **契約データ**: 7 年保持（特商法・法人税法準拠想定）
- **バックアップ**: 日次フル + 1 時間ごとの差分

## 7. Assumptions

業務要件で明示されていない事項についての作業仮説。後段エージェントが確認・更新する。

1. **ロール多重付与**: ユーザー 1 人につき卸業者ロールは 1 つに限定する（複数兼務は `wholesaler_admin` を付与して代替）。提案書 §4 で各ロールが独立して書かれているため。
2. **アポ取得組織**: 顧客・アポは「アポ取得二次店 (relationship_id)」を保持する。担当二次店と異なる場合があるため別フィールドで管理。
3. **共同開催のアポ帰属**: 共同開催イベントで獲得したアポは「アポ登録者の所属組織」を `acquired_org_type` に保持。インセンティブ分配は契約成立時に手動調整 (F-047)。
4. **自社開催由来の顧客**: 自社開催で獲得した顧客は `owner_relationship_id = null` で `wholesaler_id` 直下に置く。
5. **キャンセル期限の起点**: 「契約日」を起点とする（docs/01 §7.2 「クーリングオフ期間に準拠」より）。「契約書受領日」基準にする場合は Open Question 化。
6. **負調整の決済**: 翌月以降のインセンティブから減算するロジックは「インセンティブ確定額の合計から負調整を引いた額が支払額」とし、月をまたぐ持ち越しを許容する。
7. **共同開催の標準計算下書き**: F-046 で「卸粗利 × 関係インセンティブ率」を下書きとして提示し、F-047 で確定。
8. **二次店スコープ未上書きの扱い**: イベント単位上書きがない場合、関係デフォルト (F-010) を使用。商談ステータス変更時に判定。
9. **シフト重複チェックの範囲**: 同一卸業者テナント内で重複を検出。複数卸業者をまたぐ重複はチェックしない（自社要員は単一卸業者所属を前提）。
10. **個人情報マスキング権限**: `wholesaler_admin` は卸業者内の全顧客情報を非マスクで閲覧可。`saas_admin`（SaaS 運営者）は常にマスク表示（Open Q5 で再確認）。
11. **二次店向け契約閲覧**: 二次店は自社関与契約のみ「契約金額」「契約日」「インセンティブ対象粗利」「インセンティブ額」を閲覧可。施工費・その他原価・仕入値は不可。
12. **月次クローズの起動**: 月次集計 (F-048) は月末翌日にバッチで実行。手動再集計も可能。
13. **通知の重複排除**: 同一イベント × 同一受信者 × 同一種別の通知は 1 時間以内に重複しない。
14. **添付ファイルサイズ**: イベント報告・契約書ファイルは 1 ファイル 10MB まで、合計 50MB/契約まで。
15. **エクスポート権限**: CSV エクスポートは Phase 2 機能 (F-057) の逆方向として、`wholesaler_admin` のみに許可する想定。
16. **顧客 : 契約の多重度**: F-061 統合ビューは 1 顧客に複数契約（増設・買い替え）が存在しうる前提でタブ切替を提供する。MVP は `Deal:Contract = 1:1` を基本とし、複数契約は docs/05 §16.8-1 の確定後に拡張（既存 1:1 でも単一契約タブとして成立）。
17. **F-061 と F-062 の段階性**: 「閲覧 (F-061, P0)」を MVP 必須、「編集 (F-062, P1)」を docs/05 §16.6 の段階移行（16-B〜16-D）に沿って実装。設備明細・ローン入金の編集は新規子テーブル（`contract_equipments` / `contract_payments`）の CREATE 完了（16-C）後に有効化。
18. **設備明細の長尾属性**: 型番②・住設安心サポート・発電量保証・オプション付帯・三菱鍋型番・パワコン設置場所(新設/交換) は `contract_equipments.attributes` (Json) に格納し、F-061 は docs/05 §16.3 のキー定義に従って展開表示する。
19. **既設設備（現況）と契約後設備（`ContractEquipment`）の分離**: F-063 の既設設備は「ヒアリング時点の顧客宅の現況・クロスセル判定材料」、`ContractEquipment` は「契約成立後に当社が販売・施工する設備のスペック明細」で目的・ライフサイクル・source of truth が異なる。前者は `Customer` 子テーブル（`customer_existing_equipments`）、後者は `Contract` 子テーブルで別管理し、混在・相互参照させない。F-061 でも別カテゴリ表示。
20. **家族年齢の保持方式**: ご主人/奥様/お子様年齢は **ヒアリング時点の入力値（スナップショット）** を保持する。顧客本人年齢を `Customer.birthDate` から算出する F-061 とは別概念とし、家族については生年月日を取得しないため算出に依らない。
21. **案内者と主権者同席フラグの併存**: F-063 の `guide_attendee`（ご主人のみ/奥様のみ/ご夫婦お揃い/その他）は「誰が案内に応じたか」を表し、`faceToFace`（主権者同席の有無）とは意味が異なるため別列として併存させる。なお `faceToFace` は現状 DB（Prisma `Customer` モデル）には存在せず、デモ層（`apps/web/components/reports/customer-report-data.ts` 他）にのみ存在する項目である。F-064 が「デモ→永続」と明示するのと同様、`faceToFace` も永続化時に新 `guide_attendee` とは別概念として別列化して保持する。
22. **既設設備の機微度**: 家族年齢・家族構成・分離電話は PII としてマスキング対象。既設設備の有無＋大分類は機微度が低く既定で非マスク許容、設置日/メーカー/容量/枚数は二次店ロールに不可視（卸業者ロール限定）。pino redact + Sentry beforeSend のフィールドリストに家族年齢・分離電話を追加。
23. **レーンイベント報告の永続化単位**: F-064 はレーン (`LineEvent`) × 開催日を報告キーとし、開始/終了報告（`lane_event_reports`）と成果報告（`lane_event_results`、両面/片面内訳・種別）を別テーブルで永続化。現行デモ生成（`apps/web/components/reports/result-report-data.ts`）は永続化完了後に廃し DB を source of truth とする。項目定義はデモと一致（差分は未永続→永続のみ）。
24. **二次店希望の希望順位の意味（F-060）**: `LanePreferenceItem.priority` は「希望①②③…の優先度＝表示順」を兼ねる。MVP では優先度（卸業者が体制決定時に上位希望を優先する材料）と単なる並び順を区別せず、`priority` 昇順を希望順位として扱う。重み付けロジック（上位希望ほど採択優先など）は持たず、表示と材料提供にとどめる。
25. **希望場所のラベル一次ソース化（F-060）**: 希望場所はマスタ（場所提供元 / 店舗）リンクを任意とし、`venueLabel` 文字列を一次ソースとして必須保持する。二次店はマスタ未登録の場所も自由記述で希望提出できる。卸業者がマスタへ寄せたい場合は後からリンクを付与しても `venueLabel` は破棄しない（突合用に併存）。
26. **希望開催日の保持方式（F-060）**: 希望開催日は「週単位の複数の希望日帯」（例 7/7~8・7/12~13・7/14~15）を表現する。MVP では各日帯を `{ from: Date, to: Date }` の配列（または同等の構造化 JSON）として保持し、UI でチップ表示する。日単位フラット配列か「週＋日帯」構造かの最終粒度は Open Q26 で確定。
27. **希望レーン数の導出（F-060）**: 「希望レーン数」は運営者の管理項目には登場するが、`LanePreferenceItem` 件数から導出し DB には冗長カラムとして持たない。一覧バッジ・集計は常に件数から算出する。

## 8. Open Questions

8. **顧客電話番号の重複ルール**: 卸業者テナント内で重複検出する想定だが、複数二次店が同じ顧客を別ルートで登録した場合のマージ運用。

15. **多拠点卸業者の扱い**: 同じ卸業者法人内に複数拠点（支店）がある場合のサブテナント設計（Phase 2 以降か）。
16. **1 顧客あたりの契約数（docs/05 §16.8-1）**: 増設・買い替えを別 `Contract` とするか。F-061 統合ビューの契約タブ多重度に影響。
17. **案件ステータス enum 値（docs/05 §16.8-4）**: 架電 / 不備解消 / 完工後 / 支払の実運用選択肢を業務側で確定。F-061 のステータスバッジ表示・F-062 の編集選択肢に影響。
18. **`ContractEquipment` と `ContractItem` のリンク（docs/05 §16.8-5）**: 価格明細とスペック明細を 1:1 リンクするか独立運用か。F-061 設備明細カードの粗利連動表示に影響（粗利計算は `ContractItem` 側で完結）。
19. **既設設備ヒアリング項目の必須/任意（F-063）**: 既設ガス給湯機・エコキュート・太陽光の有無を必須化するか任意にとどめるか。現場ヒアリングの取得率に依存するため運用側で確定。
20. **家族構成の構造化粒度（F-063）**: 現行 `household` 自由記述のまま家族年齢（ご主人/奥様/お子様）を別列で持つか、世帯メンバを構造化テーブルにするか。MVP は自由記述 + 代表 3 年齢で開始。

23. **両面/片面の業務定義（F-064）**: 抽選数・着座数・アポ数の「両面/片面」が何を指すか（ブースの両面利用/片面利用等）の正式定義。デモ項目定義を踏襲しつつパイロットで確認。

25. **希望場所のマスタ連携要否（F-060）**: 希望場所ラベルからマスタ（場所提供元 / 店舗）への自動サジェスト・名寄せをどこまで実装するか。MVP は任意リンク + 手動サジェスト、自動名寄せ・表記揺れ統合は Phase 2 検討。卸業者が未リンク希望から新規レーン (F-059) を起こす運用フローも要確認。
26. **希望開催日の構造化粒度（F-060）**: 希望開催日を「日付（日帯 from-to）の配列」とするか「週インデックス + 各週内の日帯」の構造とするか。週ごと提出（例 7/7~8 / 7/12~13 / 7/14~15）の業務実態に合わせ、program-design で `desiredDates` のスキーマ（Json 構造 or 子テーブル）を確定する。
27. **再提出・履歴の扱い（F-060）**: 同一二次店 × 同一対象月の再提出を「上書き更新」とする現行方針で、提出履歴（版）を残すか最新のみ保持かを確定する。監査ログ (F-055) で変更追跡する想定だが、希望明細の差分履歴をどこまで見せるかは運用判断。
